🎯 The 3NF Rule (applied to every table)

A table is in Third Normal Form if:

  • ✅ It is already in 2NF (no partial dependency).
  • ✅ No non‑prime attribute determines another non‑prime attribute – i.e., no transitive dependency.
Formal condition for every FD X → Y:
Either X is a superkey (candidate key) OR every attribute in Y is a prime attribute (part of some candidate key).
The only forbidden pattern: non‑prime → non‑prime.

Potential violation if badly designed: If we had city → latitude (non‑prime → non‑prime), that would break 3NF.

Functional dependencies:

customer_name → customer_city (non‑prime → non‑prime) ❌
restaurant_name → restaurant_rating (non‑prime → non‑prime) ❌

❌ This table is NOT in 3NF because of transitive dependencies (customer city repeats, restaurant rating repeats).
Our fix: We split into Customers, Restaurants, and Orders. Now each fact is stored once.
📊 Summary – All 22 Tables Satisfy 3NF

✅ Eliminated transitive dependencies:

  • Order → customer city → moved to Addresses
  • Order → restaurant rating → moved to Restaurants
  • Delivery → rider phone → moved to Riders + Users
  • Address → city latitude → stored per address, no FD

✅ Every non‑prime attribute depends ONLY on:

  • the primary key (or a candidate key)
  • no chain of dependencies exists

➡️ Zero update/insert/delete anomalies.

🎯 Final check: For every functional dependency in all 22 tables, the left‑hand side is always a superkey OR the right‑hand side consists only of prime attributes. The forbidden pattern (non‑prime → non‑prime) never occurs.
