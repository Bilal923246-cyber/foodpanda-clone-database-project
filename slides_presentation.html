🔧 Normalization (3NF) – Example

  • 1NF: No repeating groups → Order_Items one row per item.
  • 2NF: No partial dependency → unit_price snapshot.
  • 3NF: No transitive dependency → Orders stores customer_id, not name.

Prevents update/insert/delete anomalies.

⚙️ SQL Server Implementation

CREATE TABLE Users (
    user_id INT IDENTITY PRIMARY KEY,
    email NVARCHAR(100) UNIQUE NOT NULL,
    user_type NVARCHAR(20) CHECK (user_type IN ('customer','owner','rider'))
);

CREATE TABLE Orders (
    order_id INT IDENTITY PRIMARY KEY,
    customer_id INT FOREIGN KEY REFERENCES Customers(customer_id),
    total_amount DECIMAL(10,2)
);

❓ Possible Teacher Questions (Click to show answer)

Why 22 tables? Could you merge some?
Each table has a single purpose. Merging would cause redundancy. For example, Order_Items is separate from Orders to allow multiple items per order (1NF).
What is a weak entity? Give an example.
Order_Items is weak – it cannot exist without an Order. ON DELETE CASCADE ensures it is removed with the parent order.
How do you enforce one‑to‑one between Orders and Payments?
By making order_id a UNIQUE FOREIGN KEY in the Payments table. This prevents multiple payments for the same order.
Write a query to find riders who never delivered an order. 
SELECT * FROM Riders WHERE rider_id NOT IN (SELECT rider_id FROM Delivery_Assignments);
How would you improve performance for large data?
Add non‑clustered indexes on foreign keys (customer_id, restaurant_id) and on frequently filtered columns like order_date and status.
